01 Folder Structure, How It Works, and What Is Being Achieved

Current Structure Overview

Synaptic is a full-stack adaptive learning platform built on Next.js 14 App Router + TypeScript + Tailwind CSS, backed by Supabase (PostgreSQL + Auth + RLS). Content lives in plain JSON files in the content/ directory and is loaded at build-time via Node require(). The adaptive engine runs entirely in TypeScript on the server — no external ML service needed. Deployment target is Vercel + GitHub.

How the System Works — Content Flow

Learner Journey

1. Onboard → /api/onboard creates learner_profiles + motivation_states rows in Supabase.

2. Diagnostic → /learn/diagnostic loads 12 questions via require(), shows them in sequence. On submit, /api/diagnostic runs placeLearner() from src/lib/graph/index.ts, initialises all 31 learner_skill_states rows, and writes SM-2 schedules.

3. Study loop → /api/session (start) calls selectNextTask() from src/lib/session/engine.ts. Engine reads skill states, SM-2 schedules, and motivation state from DB; picks the optimal next question using 4-priority selection; returns SessionTask object. Learner answers → /api/attempt verifies answer server-side, updates BKT (p_know), SM-2 interval, motivation state, and unlocks prereq graph. On correct answer, /api/explanation serves the explanation at the right depth (beginner/mid/advanced) based on p_know.

4. Explanation panel → ExplanationPanel.tsx renders body, key_insight, mini_exercise, common_mistakes, and (if present in JSON) tabs for real_world_usage, build_task, and explain_back_prompt.

5. Dashboard → Server component reads all skill states, groups by phase, renders p_know progress bars and mastery legend.

7.4 Content Guide — How to Write Missing Content

Question Writing Guide

Every question must pass this test: "If a learner gets this wrong, what exactly did they misunderstand about AI engineering?" The distractor options should reflect real misconceptions, not random wrong answers.

  • same-tier MCQ: Test the core concept at face value. One clear correct answer. Distractors should be plausible (things beginners actually believe).
  • harder-tier MCQ: Apply the concept in an AI/ML context. Numbers, code snippets, or real system scenarios. Requires understanding, not just recall.
  • review-tier fill: Complete a sentence with the key term or short phrase. Should be the thing learners most often forget. correct_answer should be 1–4 words.
  • explanation_after: Always connect to AI engineering. "This matters because in PyTorch you'll…" or "GPT-4's architecture uses this because…".

Explanation Writing Guide by Depth

beginner.json: Start with a real-world analogy (not a computing analogy). No jargon in the first paragraph. Include one simple code example (5–10 lines). End with a mini_exercise that takes 2–3 minutes and uses no imports.

mid.json: Introduce correct technical terminology. Show patterns from real ML code. Code examples 10–30 lines using numpy, pandas, or PyTorch basics. mini_exercise should require running code.

advanced.json: Production concerns, performance characteristics, trade-offs, research connections. Reference specific papers, tools (FAISS, Flash Attention, etc.), or frameworks. mini_exercise should involve a real library or profiling step.

real_world_usage Writing Guide

One paragraph, 3–5 sentences. Structure: "[Concept] is used in [specific production system / tool / company] because [technical reason]. The concrete consequence is [what engineers actually do / decide because of this]. Without understanding [concept], you would [common error or confusion]."

Always name something real: PyTorch, Hugging Face, FAISS, OpenAI API, Llama.cpp, vLLM, Weights & Biases, etc.

build_task Writing Guide

A build task should take 10–20 minutes in a Python notebook. Structure:

  • title: Action verb + short noun. "Build a Tokeniser Counter", "Implement Gradient Descent", "Create a FAISS Index".
  • context: One sentence — why a real engineer would do this.
  • steps: 4–6 concrete steps. Each step is one action. No ambiguity.
  • expected_output: Describe exactly what the learner should see when it works. Numbers, shapes, printed values.
  • hint: Reveal the smallest possible nudge. Don't give away the answer.
  • starter_code: Optional. If the setup is complex, provide imports and scaffolding with # TODO comments.

Persona Consideration

The platform serves three implicit learner types. Every question and explanation should be legible to all three:

  • Career Switcher: Has used software but never programmed. Needs strong analogies and no assumed CS knowledge. Beginner depth is their primary entry.
  • Domain Expert: Strong in their field (biology, finance, law) but new to coding. Motivate by connecting to their domain. Mid depth is their target.
  • Junior Dev: Can code in Python but has never done ML. Needs to understand why ML patterns exist, not just how they work. Advanced depth is their graduation point.
